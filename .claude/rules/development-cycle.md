# Цикл разработки задачи

## Главное правило

При `/take-task` запускается **автоматический цикл**. Сначала определи feature-size, затем запускай соответствующий пайплайн.

## Feature-size маршрутизация

| Размер | Критерии | Шаги |
|--------|----------|------|
| **S** | 1 файл, < 50 строк, нет новых зависимостей | developer → tester → tech-debt → фиксация |
| **M** | 2-5 файлов, новый модуль/компонент | analyst → developer → tester → reviewer → tech-debt → фиксация |
| **L** | 6+ файлов, архитектурные изменения | analyst → architect+reviewer(parallel) → skeptic → developer↔tester(cycles) → reviewer → tech-debt → фиксация |

## Пайплайн (полный — L)

```
/take-task [ID]
     │
     ▼
[1]  АНАЛИЗ ────── агент analyst (M, L)
     │              Читает документацию, критерии, ПМИ
     ▼
[2]  DISCOVERY ──── если есть неясности
     │              Задаёт вопросы, фиксирует решения
     ▼
[3]  ПЛАН ──────── architect + reviewer(plan_review) — ПАРАЛЛЕЛЬНО (L)
     │              Architect проектирует, reviewer проверяет план
     ▼
[4]  REALITY ───── агент skeptic (только L)
     │              Проверяет план на «миражи»
     ▼
[5]  КОД+ТЕСТЫ ── developer ↔ tester — per-feature циклы (S, M, L)
     │              Код → тесты → фикс → дальше
     ▼
[6]  РЕВЬЮ ────── агент reviewer (M, L)
     │              Финальное ревью (до 3 раундов). CRITICAL/HIGH → developer фиксит
     ▼
[7]  TECH-DEBT ── обязателен для ВСЕХ размеров (S/M/L)
     │              Зафиксировать в tech-debt.md
     ▼
[8]  ФИКСАЦИЯ ── обновить tasks.json, progress.md
     │              Предложить коммит
     ▼
[9]  CHECKPOINT ── обновить/очистить checkpoint.yml
```

## Checkpoint

При каждом переходе между шагами обнови `dev-infra/memory/checkpoint.yml`:
- `current_step` и `step_name`
- `context` — ключевые результаты
- При завершении — `active: false`

## Когда пропускать шаги

| Тип задачи | Пропустить |
|------------|-----------|
| Анализ / изучение | шаги 3-6 (нет кода) |
| Мелкий фикс (S) | шаги 1-4, 6 (analyst, plan, skeptic, review) |
| Тестирование | шаги 3-5 (нет нового кода) |
| Документация | шаги 3-6 |
| Новый модуль (L) | выполнить ВСЕ шаги |

**Важно:** tech-debt (шаг 7) обязателен для ВСЕХ размеров задач и НИКОГДА не пропускается.

## Discovery-интервью

Если требование неясно — **не додумывай**:
1. Сформулируй конкретный вопрос
2. Покажи варианты решения с trade-offs
3. Дождись ответа
4. Зафиксируй в `dev-infra/memory/decisions.md`
